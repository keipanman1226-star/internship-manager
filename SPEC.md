最終更新: 2026-07-05

このドキュメントは、別のAI/別のチャットセッションが本アプリの開発を引き継げるように、

## 1. アプリの目的

就職活動(特にインターン応募)を行う大学生が、複数の企業への応募状況・選考スケジュール・
ログイン情報を一元管理できる、個人利用のWebアプリ。就活ポートフォリオ(ES・面接でアピールする
成果物)としても使えることを意識し、実用性と見た目の両方を重視している。

## 2. ターゲットユーザー

- 就職活動中の大学生(複数社に同時並行でインターン・本選考に応募している想定)
- スマートフォンとPCの両方で使う(スマホでの操作性を特に重視)
- 開発者(ユーザー本人)がこのアプリ自体をポートフォリオとして採用担当者に見せる想定もある

## 3. 技術方針(必ず守ること)

- **HTML / CSS / JavaScriptのみ**。フレームワーク・ビルドツール・npm依存は一切使わない
- **バックエンドを持たない静的サイト**。全データはブラウザの`localStorage`に保存する
- GitHub Pagesで公開している(バックエンドなし・完全に静的配信)
- PWA対応済み(ホーム画面に追加してアプリのように使える。オフラインでも起動可能)
- コードには**初心者にもわかるような日本語コメント**を付ける(ユーザーからの明示的な要望)
- 既存機能を壊さないこと。データ形式を変える場合は必ずマイグレーション処理を書く(後述)

## 5. データベース設計(localStorage)

バックエンドがないため、`localStorage`が唯一の永続化手段。3つのキーを使用する。

### 5-1. `internships` (応募データ本体)

配列。1要素が1つの応募(企業)を表す。

```jsonc
{
  "id": "1782971853667",           // Date.now().toString() による簡易ユニークID
  "company": "〇〇株式会社",         // 企業名(必須)
  "industry": "IT・Web",            // 業界(自由入力、datalistで過去入力を提案)
  "internshipType": "データ分析",    // インターン内容(固定8分類のセレクト。下記参照)
  "deadline": "2026-07-15",         // 応募(ES)締切(必須、date input)
  "status": "ES提出済み",           // 選考状況(固定9分類。下記参照)
  "infoSessionDate": "2026-07-03",  // 説明会日(任意、date input)
  "interviewDate": "2026-07-10",    // 面接日(任意、date input)
  "internshipName": "サマーインターン2026", // インターン名(任意、自由テキスト)
  "internshipStartDate": "2026-08-20", // インターン参加開始日(任意、date input)
  "internshipEndDate": "2026-08-24",   // インターン参加終了日(任意。空なら開始日と同じ=1日だけ扱い)
  "memo": "持ち物: 筆記用具",        // メモ(自由テキスト)
  "mypageUrl": "https://mypage.example.com/login", // マイページURL(平文保存。機密でないため)
  "mypageIdEnc": { "iv": "base64...", "data": "base64..." } | null, // 暗号化されたマイページID
  "mypagePasswordEnc": { "iv": "base64...", "data": "base64..." } | null // 暗号化されたパスワード
}
```

**固定選択肢(この順序を変えない。順序がグラフ・フェーズ一覧の並びに使われる)**

- `STATUS_ORDER` (選考状況・9種類):
  `未応募 → ES提出済み → 書類選考中 → 面接予定 → 面接中 → 最終面接 → 通過 / 不合格 / 辞退`
- `INTERNSHIP_TYPE_CATEGORIES` (インターン内容・8種類、固定でこれ以外の値は存在しない):
  `データ分析, AI・機械学習, システム開発, Web開発, マーケティング, コンサル, 営業, その他`

### 5-2. `internship-manager-security` (暗号化の認証設定)

マイページID/パスワードを暗号化するための「認証方法の設定」のみを保存する。
**暗号鍵そのものは絶対に保存しない**(後述のセキュリティ設計を参照)。

```jsonc
// WebAuthn(生体認証)方式の場合
{ "method": "webauthn", "credentialId": "base64...", "salt": "base64..." }

// パスコード方式の場合(生体認証が使えない/失敗した場合のフォールバック)
{ "method": "passcode", "salt": "base64..." }
```

### 5-3. `internship-manager-custom-events` (カレンダーの自由な予定)

配列。応募企業とは無関係な、ユーザーが自由に追加した予定(大学の予定など)。

```jsonc
{
  "id": "1782973106295",
  "title": "大学のキャリアガイダンス",
  "date": "2026-07-02",
  "category": "大学",       // CUSTOM_EVENT_CATEGORIES: 面接, 説明会, ES提出, 大学, その他
  "memo": "2号館301教室"
}
```

### 5-4. データマイグレーション方針(重要・必ず踏襲すること)

`loadInternships()`は読み込み時に必ず`migrateItem()`を全件に適用し、
古い形式のデータを新しい形式に自動変換してから返す。変換後の内容が変わっていれば
即座に保存し直す。**今後スキーマを変更する際も、必ずこの仕組みで後方互換を保つこと**
(ユーザーの既存データを絶対に消さない・壊さないという要件があるため)。

これまでに行った変換の実例:
- `応募済み` → `ES提出済み`、`内定` → `通過`(選考状況名の変更)
- インターン内容の自由入力(例:「コンサルティング」「企画」)を固定8分類にマッピング
  (マッピング外は「その他」に丸める)
- 新規追加したフィールド(`infoSessionDate`, `mypageUrl`,
  `mypageIdEnc`, `mypagePasswordEnc`)が存在しない古いデータには、
  空文字列/`null`のデフォルト値を補う
- 旧`internshipDate`(1日だけの参加日)→ 新`internshipStartDate`/`internshipEndDate`
  (開始日・終了日)へ変換。旧データは「開始日=終了日=旧internshipDate」の1日だけの
  期間として引き継ぎ、`internshipDate`キー自体は変換後のオブジェクトから削除する
  (`migrateItem()`内で分割代入`const { internshipDate, ...rest } = item;`により除去)

## 6. 実装済み機能

### 6-1. 応募データのCRUD

- 右上「＋ 新規追加」ボタン、または各種一覧行のクリックから`<dialog id="form-modal">`を開く
- 追加・編集は同じフォームを共用(`editIdInput.value`の有無で分岐)
- 削除は`confirm()`で確認してから実行
- フォーム項目: 企業名(必須) / 業界 / インターン内容(セレクト) / 選考状況(セレクト) /
  応募締切(必須) / 説明会日 / 面接日 / インターン名(任意) / 参加開始日 / 参加終了日 /
  メモ / マイページ情報(下記6-6)
- インターン参加日は単一日ではなく**開始日〜終了日の期間**として管理する
  (`getInternshipRange(item)`ヘルパーで取得。終了日未入力なら開始日と同じ1日として扱う)

### 6-2. 絞り込み・検索(フィルター行)

画面上部の3つのコントロールがダッシュボード・グラフ・一覧・カレンダーすべてに連動する
(`renderAll()`が一括で再描画するため、常に整合性が取れる)。

- 企業名でのキーワード検索(部分一致・大文字小文字区別なし)
- 業界セレクト(登録データから動的に選択肢を生成)
- インターン内容セレクト(固定8分類)

※ カレンダーの自由な予定(`internship-manager-custom-events`)はこのフィルターの対象外。
　常に全件表示される(応募データと無関係な個人予定のため)。

### 6-3. ダッシュボード(7つのカード、この順番で並んでいる)

1. **今週の予定**(`renderAgenda`) — 応募企業のES締切・説明会・面接のうち、
   今日から7日以内(`days >= 0 && days <= 7`)のものだけを残り日数が少ない順に表示。
   「あと3日 **アルファ商事** インターン応募締切」のような形式。1日以内は赤、3日以内は
   オレンジで強調(`urgencyClass()`)。行をクリックすると該当企業の編集モーダルが開く。
   **自由に登録したカレンダー予定はここには含めない**(ユーザー要望による明確な役割分担)。
   インターン参加期間は特別扱いで、開始日までの残り日数が7日以内なら「あと◯日 参加予定」、
   **今日が開始日〜終了日の範囲内なら「本日参加中」を最優先(赤)で表示**する
   (`getInternshipRange()`で期間を取得し、`startDays <= 0 && endDays >= 0`で判定)。
2. **統計タイル**(`renderStatTiles`、4枚):
   - 総応募数
   - 選考中(書類選考中+面接予定+面接中+最終面接の合計)
   - ES通過率 = (面接予定+面接中+最終面接+通過) ÷ (同+不合格)。結果待ちのみの場合は「ー」
   - 通過(status === "通過" の件数)
3. **締切が近い順ランキング**(`renderRanking`) — 締切が今日以降の応募を上位5件、
   近い順に表示。1位が最も締切が近い。
4. **選考フェーズ一覧**(`renderPhaseList`) — 9つの選考状況ごとの件数と割合を横棒グラフで表示
5. **選考ファネル**(`renderFunnel`) — 応募予定(全件)→ES提出→書類通過→面接→最終面接→通過、
   の6段階のカウントダウン式ファネルと、各段階間の通過率(%)。
   **「不合格」「辞退」は書類通過以降の段階には計上しない**(どの段階で外れたか記録していない
   ための保守的な扱い。ES提出段階には含める)。
6. **業界別応募数**(`renderIndustryBreakdown`) — 件数の多い順、件数+割合を横棒グラフで表示
7. **インターン内容の分析**(`renderTypeBreakdown`) — 固定8分類ごとの件数+割合(⑥⑦は2カラムで
   並び、スマホでは1カラムに変わる)

グラフはすべて**自作の横棒グラフ(`renderBarListInto`共通関数)**。SVGの縦棒グラフは
過去バージョンで使っていたが、情報量が増えたタイミングで横棒リストに置き換えて廃止した
(モバイルでの可読性・スケーラビリティのため。詳細は「今後の改善案」参照)。

### 6-4. 応募一覧(カード表示)/ カレンダー表示(タブ切り替え)

`.view-tabs`で「一覧」「カレンダー」を切り替える(`hidden`属性で表示制御)。

**一覧**: 締切が近い順にカードを表示。各カードに企業名・選考状況バッジ・業界・内容・
締切(残り日数バッジ付き)・説明会/面接/参加日・メモ・(あれば)マイページ操作ボタン・
編集/削除ボタンを表示。

**カレンダー**: 月表示(6週間=42マス固定、日曜始まり)と週表示(7マス)を切り替えられる。
- 月表示⇔週表示の切り替えは`.calendar-mode-btn`(状態: `calendarViewMode` = `"month"|"week"`)。
  週表示の起点は`calendarWeekStart`(その週の日曜日を指すDate、`getWeekStart()`で算出)。
  切り替え時は「今表示中の月が今月なら今日の週へ、そうでなければその月の最初の週へ」ジャンプする
- 応募企業のイベント(締切=紫, 説明会=水色, 面接=青, インターン参加期間=オレンジ)を
  実線のチップで表示
- **インターン参加期間(開始日〜終了日)は、期間中の毎日にイベントを展開して連続した帯として
  表示する**(`buildEventsByDate()`が`getInternshipRange()`の期間を1日ずつ展開)。
  週の行の中で見た目がつながって見えるよう、期間の本当の開始日/終了日、または
  週の行の端(日曜列/土曜列)かどうかで角丸を制御する(`range-start`/`range-middle`/`range-end`
  クラス)。企業名のラベルは、期間の本当の初日か、行が変わって帯が新しく始まる場所にのみ表示し、
  それ以外は色だけの帯(ラベルなし)にする。**行をまたいだ帯の連続表示はしない**(週をまたぐと
  次の行でまた新しい帯として描画される。Googleカレンダー等でも一般的な簡略化)
- 自分で自由に追加した予定(面接/説明会/ES提出/大学/その他)を**点線枠**のチップで表示し、
  企業由来のイベントと視覚的に区別する
- 1日の予定件数が表示上限(月表示=2件、週表示=6件)を超える場合は「+N件」で省略表示
- **日付セル(チップ以外の空いている場所)をクリックすると、その日付が入った状態で
  予定追加モーダル(`event-modal`)が開く**
- 企業由来のチップ(締切/説明会/面接/インターン参加期間)をクリック→該当企業の編集モーダル
  (`form-modal`)が開く
- 自由な予定のチップをクリック→その予定の編集モーダル(`event-modal`、削除ボタンあり)が開く
- 前/次ボタンで月表示なら1ヶ月、週表示なら1週間移動する(`calPrevBtn`/`calNextBtn`が
  `calendarViewMode`で分岐)

### 6-5. マイページ情報の管理(URL・ID・パスワード)

- フォーム内「マイページ情報(任意)」セクションにURL・ID・パスワード欄
- **URLは平文保存**(機密情報ではないため)。IDとパスワードは個別に暗号化して保存
- 一覧の各カードに、保存されている項目に応じて以下のボタンが動的に表示される:
  - 「マイページへ移動」— `window.open(url, "_blank", "noopener,noreferrer")`
  - 「IDをコピー」— 復号してクリップボードにコピー
  - 「パスワードをコピー」— 復号してクリップボードにコピー
- **コピーは`copyTextToClipboard()`が3段階のフォールバックで試みる**:
  ①`navigator.clipboard.writeText()` → ②失敗したら`document.execCommand("copy")`
  (見えない`<input>`を一時的に作って選択してコピー) → ③それでも失敗したら
  `window.prompt()`で復号済みの値を表示し、手動で選択してコピーしてもらう。
  生体認証やパスコード入力に時間がかかると、ブラウザが「ユーザーが今操作した」と
  みなす有効期限(user activation)が切れ、特にスマホのブラウザで①②が黙って
  失敗することがあるため、③まで用意することで最終的に必ずコピーできるようにしている
  (実際にスマホでコピーできないという不具合が報告され、この対策を追加した)
- 編集モーダルでは、ID・パスワードは初期状態で伏字(`••••••••`)。
  「表示」ボタンを押すと初めて認証(生体認証 or パスコード)→復号→実際の値を表示する
- ID・パスワード欄は「実際にユーザーが入力/削除した時だけ」(`input`イベント発火で判定)
  再暗号化する。触っていなければ既存の暗号化データをそのまま引き継ぐため、
  無関係なフィールド(メモ・選考状況など)だけを編集しても認証は求められない

### 6-6. 暗号化の仕組み(セキュリティ設計・重要)

**設計思想**: バックエンドがない静的サイトである以上、真の安全性には限界があるが、
「JSのソースコードに埋め込まれた鍵で復号できる見せかけの暗号化」(セキュリティ・シアター)
は絶対に避け、実際にWeb Crypto APIによる本物の暗号化を行う。

**暗号方式**: AES-GCM(256bit)。IVは暗号化のたびに`crypto.getRandomValues`でランダム生成。

**鍵の作り方(2方式、自動フォールバック)**:
1. **WebAuthn PRF拡張(生体認証)** — `navigator.credentials.create()`に
   `extensions: { prf: {} }`を指定して生体認証(Windows Hello, Touch ID, Face ID等)の
   資格情報を作成し、PRF拡張から得られる疑似ランダム値をAES鍵として使う。
   対応していない環境では例外が発生するので、`try/catch`で検知して次にフォールバックする。
2. **パスコード方式(フォールバック)** — ユーザーが決めた文字列を`PBKDF2`
   (250,000 iterations, SHA-256)で鍵導出する。パスコード自体は保存しない。

**鍵はどこにも保存しない**。`sessionCryptoKey`という変数にメモリ上でのみキャッシュし
(`SESSION_TIMEOUT_MS = 10分`)、ページをリロードすれば消える。復号に失敗した場合
(パスコード間違い)は**即座にキャッシュを破棄する**(でないと再入力の機会がないまま
失敗し続けるバグになる。実際に開発中に発見し修正した)。

**全体の関数構成**(`script.js`内、この順で呼び出される):
`ensureUnlocked()` → (未設定なら`setupSecurity()`→WebAuthn試行→ダメなら`askPasscode({mode:"setup"})`) →
(設定済みなら`config.method`に応じて`getWebAuthnKey()`または`askPasscode({mode:"unlock"})`) →
`encryptMypageField()` / `decryptMypageField()`

### 6-7. PWA対応

- `manifest.json`: アプリ名・アイコン・`display: standalone`・テーマカラー
- `sw.js`: **ネットワーク優先、失敗時のみキャッシュにフォールバック**する方式
  (過去にキャッシュ優先方式で「更新してもスマホに反映されない」不具合が発生し、
  ネットワーク優先に変更した経緯がある。**この方式を維持すること**)
- アイコン4種(192px, 512px, apple-touch-icon 180px, favicon 32px)

```
  <div id="toast">                              … コピー完了等の通知
```

## 8. 各画面のUI/UX要件

### 全体共通
- モバイルファースト。ブレークポイントは`480px`(`@media (max-width: 480px)`)
- カードサイズ・余白は`.dashboard-card`, `.stat-tile`, `.card`で統一
- モーダルはネイティブ`<dialog>`要素を使用。`showModal()`/`close()`で開閉。
  **`dialog#xxx { display: flex }`のように無条件でdisplayを指定すると、
  `hidden`状態でもCSS詳細度の関係で表示されてしまうバグを過去に踏んだ。
  必ず`dialog#xxx[open]`セレクタの中でのみdisplayを指定すること**
- `hidden`属性は`[hidden] { display: none !important; }`をグローバルに定義して、
  他のクラス(`display: flex`等)と衝突しても必ず非表示になるようにしている

### ダッシュボード
- 情報量を増やしつつ、締切が近いものは赤/オレンジで必ず目立たせる(`urgencyClass`)
- 横棒グラフは`renderBarListInto()`共通関数で統一(見た目・実装ともに一本化)

### 一覧/カレンダー
- カードは`grid-template-columns: repeat(auto-fill, minmax(270px, 1fr))`でレスポンシブ
- 月表示は常に6週間(42マス)固定表示(月によって高さが変わらないようにするため)。
  週表示は7マスで、月表示より`min-height`を大きく取り、1セルあたりの表示件数上限も増やす
- カレンダーセルは「タップで予定追加」ができることを`cursor: pointer`とヒントテキストで示す
- チップクリック時は必ず`event.stopPropagation()`でセルへのバブリングを止める
  (でないと、チップをタップしたのに背後のセルの「予定追加」も同時に発火してしまう)
- インターン参加期間の帯チップは、ラベルなし(空文字)になることがあるため、
  `.calendar-chip`に`min-height`を明示指定している(指定がないと、文字が空の時に
  高さが4px程度まで潰れて帯がほぼ見えなくなるバグになる。実際に発生し修正した)

### モーダル
- ヘッダー(タイトル+×ボタン)/ボディ(スクロール可)/フッター(ボタン)の3段構成で統一
- 送信中は多重送信防止のため送信ボタンを`disabled`にする(特に暗号化で認証待ちが発生する
  マイページ情報の保存で重要。実際に競合状態のバグを踏んで修正した)

## 9. デザイン方針

- **配色**: Notion / Stripeを意識した「白基調 + アクセント1色」
  - ページ背景 `--page-bg: #f9f9f7` / カード面 `--surface: #fcfcfb`
  - アクセントカラー(紫寄りの青) `--accent: #4a3aa7`
  - 文字色は3段階(`--text-primary` / `--text-secondary` / `--text-muted`)
- **選考状況の色**は`STATUS_COLORS`(JS)と`--status-*`(CSS変数)の二重管理になっている。
  **両方を必ず同時に更新すること**(片方だけ変えると色がズレる)
- **警告色**: 赤(`--danger` / `--danger-bg`)= 1日以内・締切超過、
  オレンジ(`--warning-bg`)= 3日以内
- フォント: `-apple-system, BlinkMacSystemFont, "Segoe UI", "Hiragino Kaku Gothic ProN", "Yu Gothic", Meiryo, Roboto, sans-serif`
- 角丸: `--radius-lg: 16px`(モーダル) / `--radius-md: 12px`(カード) / `--radius-sm: 8px`(ボタン・入力欄)
- ロゴマークは"IM"の2文字(絵文字は使用しない。ユーザーからの指示なし)

## 10. 追加予定機能 / 今後の改善案

以下は会話中に検討したが**未実装**の項目。優先度は開発者(ユーザー)の判断に委ねる。

   状態にしたい場合に追加する。
3. **ダークモード** — 未検討・未実装。
4. **カレンダーの自由な予定にリマインダー通知機能** — Web Push等が必要になり、
   バックエンドなしでは制約が大きいため要検討。
5. **選考ファネルの段階定義の見直し** — 現在「書類通過」と「面接」を区別するのに

## 11. 開発時の注意事項(必読)

1. **既存データを壊さない**。フィールド追加・選考状況名の変更などは、必ず`migrateItem()`に
   変換ロジックを追加すること。既存ユーザーの`localStorage`には過去の形式のデータが
   残っている前提で実装する。
2. **STATUS_ORDER / STATUS_COLORS / CSS変数`--status-*`は3箇所を必ず同時に更新する**。
   インターン内容の`INTERNSHIP_TYPE_CATEGORIES` / `TYPE_COLORS`も同様(2箇所)。
3. **`hidden`属性とCSSの`display`指定が競合しないよう注意**。`[hidden] { display: none !important; }`
   のグローバルルールを削除しないこと。
4. **`<dialog>`要素のCSSは必ず`dialog#xxx[open]`セレクタの中でdisplayを指定する**
   (無条件指定はCSS詳細度の罠で閉じているのに表示されるバグになる)。
5. **フォームの送信ハンドラで非同期処理(暗号化など)を挟む場合、フォームの値は
   `await`より前に同期的に読み取ってローカル変数に保存しておく**。認証待ちの間に
   モーダルが再利用される可能性があり、後から`input.value`を読むと値が変わっている
   ことがある(実際に踏んだバグ)。また多重送信防止のため送信ボタンを`disabled`にする。
6. **カレンダーのセルクリックとチップクリックが両方発火しないよう、
   チップのクリックハンドラで必ず`event.stopPropagation()`を呼ぶ**。
7. **暗号化の復号に失敗したら、キャッシュ済みの鍵(`sessionCryptoKey`)を即座に
   破棄する**。破棄しないと、パスコードを間違えた後に再入力する手段がなくなる。
8. **sw.jsはネットワーク優先方式を維持する**(キャッシュ優先に戻すと、更新が
   スマホに反映されない不具合が再発する)。
9. Gitリポジトリは`c:\Users\keipa\Downloads\internship-manager`にあり、
   `main`ブランチに直接コミット→pushしている。GitHub CLI(`gh`)でリポジトリを
   作成済み・GitHub Pagesを有効化済み。デプロイはpush後、数十秒〜1分程度で
   `https://keipanman1226-star.github.io/internship-manager/`に自動反映される。
   コミットメッセージは日本語で、変更内容を簡潔にまとめる方針で統一している。
10. **開発環境にはnode/npm/git/ghは当初入っておらず、winget経由でGitとGitHub CLIを
    インストール済み**。ビルドツールは使わないため、node自体は本番運用に不要
    (テスト・デプロイ作業のための一時的なツールとして導入した)。
11. **テストは実際にブラウザ(headless Chrome + CDP)を操作して行う方針**。
    フォームへの値設定は`el.value = x`だけでなく`el.dispatchEvent(new Event('input'))`
    を必ず併用しないと、`input`イベントリスナーに依存するロジック
    (`mypageIdChanged`等)が反応しない。
12. **クリップボードへのコピーは、認証(生体認証/パスコード)を挟んだ後に
    `navigator.clipboard.writeText()`を呼ぶだけでは、スマホのブラウザで
    黙って失敗することがある**(認証待ちの間にuser activationの有効期限が切れるため)。
    `copyTextToClipboard()`のように、execCommand・`window.prompt()`への
    フォールバックを必ず用意すること。
13. **同じ見た目にしたいUIパーツでも、`querySelectorAll()`で拾われるクラス名を
    絶対に使い回さないこと**。月表示/週表示の切り替えボタンに、一覧/カレンダーの
    タブと見た目を揃えるつもりで同じ`.view-tab`クラスを付けたところ、
    `viewTabs = document.querySelectorAll(".view-tab")`が意図せず両方を拾ってしまい、
    月表示ボタンを押すと一覧/カレンダーの表示状態まで壊れるバグになった(実際に発生し修正)。
    見た目を共有したい場合はCSS側で複数セレクタに同じルールを適用し(`.view-tab, .calendar-mode-btn { ... }`)、
    JS側で拾うクラス名は機能ごとに別にすること。

## 12. これまでの主要な意思決定の経緯(背景理解のため)

- ユーザーは「就活生向けポートフォリオ」として本アプリを位置づけており、機能性と
  見た目の両方でアピールできることを重視している。
- スマホでの利用を強く意識しており、機能追加のたびにモバイル表示の確認を行っている。
- セキュリティ機能(マイページ情報の暗号化)については、「本物のセキュリティ」を
  求められ、見せかけの実装(base64エンコード等)は明確に拒否する方針で進めた。
  ただし「バックエンドがない静的サイトである」という limitation は都度ユーザーに
  明示している。
- 段階的に機能を拡張してきた経緧: ①基本CRUD → ②モダンUI刷新+ダッシュボード/グラフ+PWA
  → ③ダッシュボード大幅拡張(今週の予定/ランキング/フェーズ/ファネル/業界・内容分析) →
  ④マイページ情報管理(暗号化) → ⑤カレンダー機能(自由な予定)追加+今週の予定の役割明確化 →
  ⑥インターン参加日を単一日から期間(開始日・終了日)に変更し、カレンダーに連続表示・
  月表示/週表示の切り替えを追加、今週の予定に「本日参加中」の特別強調を追加。
  今後も同様に、既存機能への影響を都度確認しながら進めること。
